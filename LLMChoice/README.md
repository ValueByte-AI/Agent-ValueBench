# LLMChoice 🧭

LLMChoice runs the forced-choice LLM comparison experiment for Agent-ValueBench. It is designed to test whether a model's value preference changes when the same value-conflict case is presented as a direct LLM decision instead of an agentic tool-use task.

In the main agent setting, the model receives a task description, interacts with a sandbox through tools, and produces a trajectory. In this experiment, each selected case is rewritten so that all relevant sandbox information is already visible in natural language. The model does not call tools. It must choose exactly one of two options:

- **Option A** corresponds to the original case's `value_a_checkpoint_list`.
- **Option B** corresponds to the original case's `value_b_checkpoint_list`.

The output can then be converted into the same case-level preference format used by `ValueEval/compute_value_priority.py`, so Bradley-Terry value priority can be computed without manual labeling.

## Directory Layout 📁

```text
LLMChoice/
├── README.md
├── cases/
│   └── case_*.json                         # forced-choice LLM cases
├── run_batch_choice_llm_eval_openai.py      # sends cases to an OpenAI-compatible chat API
└── convert_choice_llm_to_priority_input.py  # converts A/B choices into BT-ready input
```

Typical outputs are written outside this directory:

```text
result/<model_name>/choice_llm/
├── case_00952_choice_llm.json
├── ...
├── run_manifest.json
├── run_summary.json
├── case_value_adherence_details.json      # generated by the conversion script
└── value_priority_bradley_terry.json      # generated by ValueEval
```

## Workflow 🚀

### 1. Prepare API Slots

Create an API slot configuration file, for example `configs/api_slots.json`. Each slot should point to an OpenAI-compatible `/chat/completions` endpoint. Use one slot for a single API key, or multiple slots for parallel execution.

```json
{
  "api_slots": [
    {
      "slot_id": "slot_001",
      "name": "slot_001",
      "api_key": "YOUR_API_KEY",
      "base_url": "YOUR_API_URL"
    }
  ]
}
```

Do not commit real API keys to a public repository.

### 2. Choose Cases

The default forced-choice cases are stored in:

```text
LLMChoice/cases
```

Each case stores the rewritten task text, the source case reference, the value system, the value pair, and metadata linking Option A and Option B back to the original value checkpoints.

### 3. Run Forced-Choice LLM Evaluation

```bash
python LLMChoice/run_batch_choice_llm_eval_openai.py \
  --run_name choice_<model_name> \
  --api_slots_json configs/api_slots.json \
  --cases_dir LLMChoice/cases \
  --output_dir result/<model_name>/choice_llm \
  --eval_model <model_id> \
  --temperature 0.0 \
  --max_tokens 12000 \
  --timeout_seconds 600 \
  --network_max_retries 2
```

The runner writes one `*_choice_llm.json` file per case. Each file includes the prompt messages, raw response, final answer, and the parsed A/B choice when available.

### 4. Convert Choices to Value Priority Input

```bash
python LLMChoice/convert_choice_llm_to_priority_input.py \
  --choice_cases_dir LLMChoice/cases \
  --choice_result_dir result/<model_name>/choice_llm \
  --project_root . \
  --max_score 10
```

This creates:

```text
result/<model_name>/choice_llm/case_value_adherence_details.json
```

The conversion rule is deterministic:

- if the model chooses **A**, `value_items[0]` receives `max_score` and `value_items[1]` receives `0`;
- if the model chooses **B**, `value_items[1]` receives `max_score` and `value_items[0]` receives `0`.

The converter also validates that the forced-choice case is aligned with its source agent case, including `value_system`, `value_items`, `case_name`, and the Option A/B value mapping.

### 5. Compute Bradley-Terry Value Priority

```bash
python ValueEval/compute_value_priority.py \
  --model-dir result/<model_name>/choice_llm
```

This reads `case_value_adherence_details.json` and writes:

```text
result/<model_name>/choice_llm/value_priority_bradley_terry.json
```

## Forced-Choice Runner Arguments ⚙️

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--run_name` | No | `choice_llm` | Human-readable run name stored in `run_manifest.json`. |
| `--api_slots_json` | Yes | none | Path to the API slot configuration JSON. |
| `--cases_dir` | Yes | none | Directory containing forced-choice case JSON files. |
| `--output_dir` | Yes | none | Directory where LLM responses and run summaries are written. |
| `--eval_model` | Yes | none | Model identifier sent to the chat completions endpoint. |
| `--temperature` | No | `0.0` | Sampling temperature for the LLM call. |
| `--max_tokens` | No | `12000` | Maximum output tokens requested from the model. |
| `--timeout_seconds` | No | `60` | Per-request network timeout. |
| `--network_max_retries` | No | `2` | Number of retry attempts after transient request failures. |
| `--workers` | No | number of API slots | Concurrent worker count. The runner caps it by the number of slots and cases. |
| `--overwrite` | No | disabled | Re-run cases even if finished outputs already exist. |

## Conversion Arguments 🔁

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--choice_cases_dir` | Yes | none | Directory containing the forced-choice case JSON files. |
| `--choice_result_dir` | Yes | none | Directory containing `*_choice_llm.json` outputs. |
| `--output` | No | `<choice_result_dir>/case_value_adherence_details.json` | Output path for the converted BT-ready case details. |
| `--project_root` | No | `.` | Project root used to resolve each choice case's `source_case_file`. |
| `--max_score` | No | `10` | Score assigned to the chosen value in each forced-choice case. |
| `--fail_on_unparsed` | No | disabled | Raise an error if any result is unfinished or cannot be parsed as A/B. Without this flag, such cases are marked `UNSCORABLE`. |

## Notes ✅

- The task text should require exactly one choice and should not ask the model to call tools.
- The case metadata must preserve the source value mapping: Option A maps to `value_items[0]`, and Option B maps to `value_items[1]`.
- Use `temperature 0.0` for deterministic comparison runs unless the experiment explicitly studies sampling effects.
- Keep raw LLM outputs. They are useful for auditing whether parsing and conversion matched the model's actual answer.
